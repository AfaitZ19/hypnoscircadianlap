03 — Bridge App
Bridge App — ทำงานอย่างไร?
ℹ️
Bridge คืออะไร: โปรแกรมเล็กๆ (~5–10 MB) ที่ติดตั้งบนเครื่อง Windows/Mac ของเจ้าหน้าที่หรือ Sales ทำหน้าที่ watch SD Card และส่งข้อมูลขึ้น cloud อัตโนมัติ ตัว UI ทั้งหมดยังอยู่บน web browser
1️⃣
SD Card Detection
auto-detect on insert
  • Bridge รันเป็น background service (startup)
  • ตรวจ drive ใหม่ทุก 2 วินาที
  • เมื่อพบ SD Card → scan หาไฟล์ CPAP ที่รู้จัก
  • แสดง notification: "พบข้อมูล CPAP — กดเพื่อ upload"
Node.js / Electron chokidar (file watch)
↓
2️⃣
ระบุ Brand & Parser
auto-detect device brand
  • ตรวจ folder structure บน SD Card → ระบุแบรนด์
  • APEX: มีโฟลเดอร์ DATALOG/ + ไฟล์ .SAP
  • Beyond: มีโฟลเดอร์ RECORD/ + ไฟล์ .bin
  • ResMed: มีโฟลเดอร์ SETTINGS/ + STR.edf
  • เรียก parser ที่ตรงกัน
APEX parser — พร้อมแล้ว Beyond — ต้อง reverse ResMed — ซับซ้อน
↓
3️⃣
Parse & Normalize
แปลงเป็น standard JSON
  • อ่าน binary → แปลงเป็น session records
  • Normalize เป็น format กลาง (ไม่ขึ้นกับแบรนด์)
  • คำนวณ AHI, compliance %, 95th percentile pressure
  • แนบ metadata: device serial, firmware, date range
output: JSON
↓
4️⃣
ยืนยันตัวตน + เลือกคนไข้
match patient before upload
  • Bridge เปิด mini UI ใน browser
  • แสดง preview ข้อมูล: กี่ session, date range, device
  • ค้นหาคนไข้ในระบบ (ชื่อ / HN / serial เครื่อง)
  • ยืนยัน → กด "Upload"
ป้องกันโหลดข้อมูลผิดคน
↓
5️⃣
Upload ขึ้น Cloud
HTTPS → API
  • POST JSON payload ไปยัง api.skylinemedical.co.th
  • JWT token ของ uploader แนบใน header
  • รองรับ offline: ถ้าไม่มีเน็ต → queue ไว้ upload ทีหลัง
  • แสดง progress bar + ผลลัพธ์
incremental sync (ไม่โหลดซ้ำ)
04 — Parser
SD Card Parser — แต่ละแบรนด์
Brand / Device Format Folder Structure สถานะ ความยาก
APEX iX Auto .SAP binary DATALOG/YYYYMMDD.SAP ✓ พร้อมแล้ว —
APEX XT Auto .SAP binary คล้าย iX Auto ต้องทดสอบ ⭐⭐
Beyond ResFree R-20i .bin binary RECORD/ ต้อง reverse engineer ⭐⭐⭐
Beyond ResFree R-30F .bin binary RECORD/ ต้อง reverse engineer ⭐⭐⭐
ResMed AirSense 10/11 EDF / STR.edf SETTINGS/ + DATALOG/ ซับซ้อน / licensed ⭐⭐⭐⭐
Philips DreamStation Proprietary — ยากมาก ⭐⭐⭐⭐⭐
⚠️
Beyond .bin format: ต้องได้ SD Card จริงเพื่อ reverse engineer — ถ้า Beyond Medical ให้ documentation ได้จะเร็วกว่ามาก ควรถามใน business meeting ครั้งหน้า
05 — Cloud API
Cloud API — Endpoints หลัก
Endpoint Method ใครเรียก ทำอะไร
/cpap/upload POST Bridge รับ session data JSON จาก bridge + save ลง DB
/patients GET Bridge / Portal ค้นหาคนไข้ (ชื่อ / HN / serial)
/patients/:id/sessions GET Portal (หมอ) ดู session history ของคนไข้คนนั้น
/patients/:id/summary GET Portal (คนไข้) สรุป AHI, compliance เฉลี่ย 30/90 วัน
/hospitals/:id/overview GET Portal (หมอ / admin) dashboard ภาพรวม รพ.
/admin/dashboard GET Skyline ทุก รพ. รวมกัน, fleet overview
/auth/login POST ทุก role JWT token

Hospital Admin

📊

09 — Risks
ความเสี่ยงและสิ่งต้องระวัง
⚠️
Beyond .bin format ไม่เปิดเผย: ต้องใช้เวลา reverse engineer หรือขอ documentation จาก Beyond Medical โดยตรง — ควรถามในการประชุมครั้งหน้า
⚠️
PDPA (พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล): ข้อมูลสุขภาพเป็น sensitive data — ต้องมี consent form, data retention policy, และ security audit ก่อน go-live กับ รพ. จริง
⚠️
Patient matching: ถ้า match ผิดคน = ข้อมูลผิดคนไข้ — ต้องออกแบบ UI ให้ยืนยันชัดเจนก่อน upload ทุกครั้ง
💡
ข้อได้เปรียบของ Skyline: เป็นเจ้าเดียวที่มี APEX SAP parser ที่ทำงานได้จริง — ถ้าพัฒนา platform นี้ได้ จะเป็น differentiator ที่แข่งขันได้ยากมากในตลาด รพ. ไทย
✅
สิ่งที่มีอยู่แล้ว: APEX SAP parser, Skyline Plus API infrastructure (.NET), JWT auth pattern, patient/device data model — Phase 1 เริ่มได้เลยโดยไม่ต้อง build from scratch
